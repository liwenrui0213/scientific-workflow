# Repository Instructions

## Mathematical communication

Define every nonstandard symbol before use. Explain formulas and principles with mathematically rigorous, logically self-contained reasoning.

## Claim-to-Evidence scientific workflow

- Treat `scientific-workflow/repository-profile.json` as the repository adaptation contract. Validate it before assuming Study, output, source, test, experiment, working-directory, or validation-command paths.
- Keep the kernel minimal: a mutable Workspace outside the cognitive and control graphs, plus one logical ledger family for occurrence and integrity. The Run ledger and Observation, Evidence, Confirmation, Checkpoint, graph-record, and Review/Verdict sequences have distinct roles; none creates scientific support. Complete-inventory sequences make finalized authority omission-evident. Recover only one uniquely interrupted publication through its explicit forward command; never reconstruct a lower history. Use the explicit migration command only for a genuinely pre-sequence Study.

### Study Skill routing

| User request | Required route |
|---|---|
| One-off scientific discussion, explanation, or brainstorming | Answer directly. Do not create or modify a Study. |
| Explicit request to start, create, or persistently investigate a new scientific question | Use `start-scientific-study`; create one draft and stop at Brief approval. |
| Named existing Study ID | Run `python -m tools.studyctl resolve-study STUDY_ID`. Use `start-scientific-study` to revise the same unapproved draft, `scientific-study` for a fresh approved Brief, and report missing or invalid state instead of creating a replacement. |
| ID-less request to continue or resume previous/current research | Run `python -m tools.studyctl resolve-study`. Follow its unique draft or approved route. On zero, multiple, or invalid candidates, ask once and never initialize a new Study. |

- A Verdict records human interpretation; it does not by itself close a Study or remove it from ID-less resolution.
- Align just in time: inspect and draft before asking; ask only when a material ambiguity has no safe reversible default. At one decision boundary, ask one batch of at most three questions and never repeat the same unresolved question.
- Begin research or execution only after `python -m tools.studyctl validate STUDY_ID` and from generated `ACTIVE_CONTEXT.json`. `context` itself revalidates and refuses to replace the last valid projection when authority is inconsistent. It provides bounded Claim and Frontier locators, resumable work, and a deterministic occurrence locator that separates nonterminal `in_progress` Runs from terminal failed/interrupted/incomplete Runs, missing outputs, ineligible attempts, and finalized-but-undispositioned Evidence. Evidence already dispositioned by an immutable archived Claim stays dispositioned after that Claim is pruned from the active file. Mutable graph drafts appear only under `workspace.graph_record_drafts`; finalized locators appear under `graph_records`. Resume before replacing, and read authority by question or ID; projections and derived Checkpoints are not scientific truth.
- Keep `work/active/` as mutable Workspace outside both graphs; move adopted code, configurations, and tests to profile host roots, and never base a Final Claim only on Workspace material. ExperimentIntent records durable **why** and may bind directly to a Run without a PLAN; ControlGraphSpec/active PLAN records **how** only when formalization or real control complexity warrants it. The Agent chooses topology within protected boundaries. `studyctl` validates bindings but neither interprets opaque control policy nor executes the whole graph. A planned Run binds only when explicitly requested and must match the exact finalized graph, Intent, and node; an active PLAN never binds or enters the formal snapshot of an ordinary Run implicitly. Activate and deactivate PLAN only through their commands; both transitions append immutable lifecycle events, and deactivation removes only the reconstructable materialized pointer. Finalized graph records and lifecycle events remain sequence-backed and append-only.
- Before modifying host code or tests, use the Study branch and optional linked-worktree policy from the profile, then create `formal/CHANGESET.json` with `python -m tools.studyctl changeset-new STUDY_ID ...`. The actual Git diff is authoritative; self-reported paths never grant write permission. Workflow enforcement code and configured protected paths are never Study outputs.
- Commit allowlisted host code, tests, and experiment assets; run `python -m tools.studyctl validate-changes STUDY_ID` to execute the profile's native validation commands and seal `formal/VALIDATION.json`; then require `python -m tools.studyctl check-changes STUDY_ID` to pass before an Evidence-producing Run. Renew a stale base anchor only through `python -m tools.studyctl changeset-renew STUDY_ID`.
- Before expensive, shared, scientifically consequential, or difficult-to-reverse work, run `python -m tools.studyctl check-formalization STUDY_ID` with the applicable flags. Create only the smallest required formal artifact.
- Treat completed Runs as immutable. Keep observations inline unless the exact versioned Registry trigger applies. A new trigger requires independent review, explicit human adoption, and protected maintenance. A finalized Observation binds its Registry, source Runs, Intent refs derived from each Run's independent `intent_binding`, and analysis, but has no Claim assessment.
- Each Evidence Argument addresses one exact Claim statement/scope, embeds its observation or pins one finalized Observation, and binds Run-derived Intent refs. It must state the reasoning bridge, auxiliary assumptions, competing explanations, and falsifiers; a result summary is insufficient.
- Runs are exploratory by default. Freeze a minimal Confirmation only before high-strength promotion; each slot predeclares acceptable terminal statuses and required Observation/Artifact outputs, and only new Runs bound to its exact protected fields and unused slots are confirmatory. One exact Claim version has one active derived campaign at a time whose Evidence discloses every record and attempt. Finalized Confirmations and whole-campaign abandonments share one complete-inventory sequence; deletion, replacement, broken links, and unindexed tails fail closed, and only the explicit one-record forward recovery may advance an interrupted finalization. Only an explicit user instruction may append a whole-campaign abandonment; it preserves every record, slot, Run, and Evidence, forbids selective slot abandonment and further Run admission, removes the campaign from confirmatory support, and may authorize a new digest-linked campaign with an explicit restart rationale. New Run/supporting-Evidence admission must replay that exact chain and select its unique active tail. Pre-transition drafts become non-resumable and require an explicit Workspace disposition before replacement. Never relabel old Runs.
- Put declared outputs only at new paths below the ignored `object_root`; declare mutable or external inputs. Missing output is recorded absent and makes the Run Evidence-ineligible without erasing its attempt, logs, budget, or history. Dirty/unverifiable scope, stale proof, changed dependencies, or altered snapshots likewise bar Evidence use.
- Preserve contradictions and representative failed, interrupted, or incomplete Runs. Run is fact, cause is hypothesis, and a reusable lesson is an Evidence-backed scoped Claim. A free-form “failed direction” note is not authority; never silently combine incompatible Cohorts.
- Exploratory Evidence may support `under_test` or scoped `partially_supported` Claims. A `numerically_supported` Claim requires non-empty bounded scope, complete current-campaign confirmatory Evidence whose supporting Runs satisfy frozen outcome contracts, and either workflow-observed fresh held-out conditions or an explicitly justified `not_applicable` condition. Contradictory Evidence requires a resolved or scope-limited synthesis; mixed Evidence must expose both parts and cannot hide omitted attempts.
- Treat `status` pressure as deterministic: compact at soft pressure before discretionary growth; at hard pressure, compact before another Run, Evidence draft, or review. Compaction remains available and deletes no history. Its Checkpoint is a source-bound derivative for resumption/review, not Evidence.
- Never silently change the Brief, protected conditions, evaluator, data split, acceptance criteria, hard budget, Claim scope, or final scientific interpretation.
- Runs and Artifacts are control/provenance facts, not cognitive conclusions. Control-graph completion is not scientific support: never update a Claim directly from a successful Plan or Run; form an Observation, write Claim-specific Evidence, and disclose the reasoning.
- The implementer is not the final reviewer; use a fresh top-level read-only session. Importing Review archives exact Review and packet bytes; Verdict binds their digests, and the shared sequence binds every Review import and Verdict occurrence. Any absent or stale Review requires a separately explicit human waiver with its own authorization text and reason; an interactive decision file must still obtain the waiver phrase in that terminal session. The human owns Verdict decisions. Only an explicit instruction adopting the exact implementation/scientific decisions, Claim scope, rationale, and conditions lets a separate write-enabled Agent record them. Silence, review success, generic continuation, or Agent advice is not acceptance.
